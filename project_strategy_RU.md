## ЧАСТЬ I — Структурный Анализ и План Проекта

---

### 1. Время Блоков в EVM-сетях и Расчёт Временных Окон

Каждая EVM-совместимая сеть производит блоки с фиксированной периодичностью. Поскольку один API-запрос, как правило, возвращает один блок, количество блоков в заданном временном окне напрямую определяет необходимое число API-вызовов для каждой сети.

#### Время блока по сетям

| Сеть | Время блока | Примечание |
|---|---|---|
| Ethereum | ~12 секунд | Фиксированное время слота после перехода на PoS (Merge) |
| BNB Smart Chain | ~0.75 секунды | После хард-форка Maxwell (ранее 3с); обновление Fermi нацелено на 0.45с |
| Polygon PoS | ~2 секунды | Стабильно на протяжении последнего времени |
| Avalanche C-Chain | ~2 секунды | Практическое среднее; при низкой нагрузке может быть ниже |

#### Количество блоков за период наблюдения

| Окно | Ethereum (12с) | BSC (0.75с) | Polygon (2с) | Avalanche (2с) |
|---|---|---|---|---|
| 1 день | ~7 200 | ~115 200 | ~43 200 | ~43 200 |
| 10 дней | ~72 000 | ~1 152 000 | ~432 000 | ~432 000 |
| 100 дней | ~720 000 | ~11 520 000 | ~4 320 000 | ~4 320 000 |

**Ключевое наблюдение:** BSC генерирует значительно больше блоков в сутки, чем любая другая из рассматриваемых сетей, за счёт суб-секундного времени блока. При наивном подходе — запросе каждого блока по порядку — BSC потребует наибольшего числа API-вызовов. Практическое решение: выборка блоков с шагом (sampling) или использование batch-запросов (`eth_getBlockByNumber` с диапазонами).

### 2. Рекомендуемые Сети — Скорость Получения Данных

Критерий выбора: **минимальное время от отправки запроса до получения пригодных данных**, с учётом как задержки ответа API, так и общего объёма данных для обработки.

Здесь есть контринтуитивный компромисс: чем быстрее генерируются блоки (BSC), тем больше блоков в сутки и тем больше API-вызовов и вычислений. Медленные сети (Ethereum) означают меньше блоков для сканирования. «Наибыстрейшее получение данных» зависит от того, что оптимизируется: число API-запросов или полнота выборки.

#### Рекомендованный набор: Ethereum, Polygon, BNB Smart Chain

**Ethereum** рекомендован из-за наименьшего числа блоков в сутки (~7 200) — полные данные за один день можно собрать минимальным числом вызовов. Все крупные провайдеры (Alchemy, Infura, Ankr) поддерживают Ethereum в качестве основной и наиболее отлаженной сети. Время ответа RPC на Ethereum — самое низкое и стабильное среди всех провайдеров.

**Polygon PoS** рекомендован за 2-секундное время блока (43 200 блоков/сутки — вполне управляемо), отличную поддержку у Alchemy и Infura, а также бесплатный API PolygonScan с лимитом 5 запросов/с. Polygon также отличается высоким объёмом EOA-транзакций, что делает выборку статистически богатой.

**BNB Smart Chain** рекомендован потому, что профессор прямо назвал его самым щедрым по бесплатным лимитам (10 000 запросов за 5 минут на BscScan). Кроме того, BSC имеет наибольший суточный объём транзакций среди EVM-сетей (12M+ txs/день), обеспечивая самую большую выборку. Компромисс большого числа блоков компенсируется щедрыми лимитами и возможностью эффективной пакетной обработки.

**Avalanche C-Chain** может быть добавлена как опциональная четвёртая сеть. Время блока такое же, как у Polygon (~2с), но суточный объём транзакций ниже — это означает меньше данных на блок и более быстрые ответы.

### 3. Бесплатные Лимиты API по Провайдерам

Перечислены рекомендуемые для проекта провайдеры (Etherscan исключён согласно условиям).

#### BscScan

Бесплатно при наличии API-ключа. Лимит: **5 запросов/с**, до **100 000 запросов/сутки**. До 10 000 записей на один вызов для эндпоинтов со списком транзакций. Лучше всего подходит для точечных запросов по адресу (например, получение всех транзакций конкретного адреса). Не оптимален для поблочного сканирования — для этого следует использовать BNB RPC.

Примечание: BscScan находится в процессе миграции — разработчики могут быть перенаправлены на Etherscan API V2 или BSCTrace. Актуальный статус: docs.bscscan.com.

#### PolygonScan

Бесплатно при наличии API-ключа. Лимит: **5 запросов/с**, до **100 000 запросов/сутки**, максимум **1 000 записей в ответе**. Отлично подходит для запросов на уровне адреса в Polygon (`?module=account&action=txlist&address=0x...`). Для получения сырых данных блоков — использовать Alchemy или Infura.

#### Alchemy

Бесплатный тариф: **300 миллионов Compute Units (CU) в месяц** при пропускной способности ~330 CU/с. Примерная стоимость методов: `eth_getBlockByNumber` (полные транзакции) ≈ 50 CU, `eth_getCode` ≈ 26 CU. Итого: порядка **6 миллионов полных блоков/месяц** на бесплатном тарифе. Поддерживает Ethereum, Polygon, BSC, Arbitrum, Avalanche, Base и 30+ других сетей с одним API-ключом. **Лучший провайдер для получения сырых данных блоков в данном проекте.**

#### Infura

Бесплатный тариф: **100 000 запросов/сутки** (кредитная модель — более тяжёлые методы расходуют больше кредитов). Поддерживает Ethereum в качестве основной сети, а также Polygon, BSC, Avalanche, Linea и другие. Хорошо подходит в качестве резервного провайдера. Более простая ценовая модель по сравнению с Alchemy.

#### Ankr

Публичные эндпоинты (без регистрации) доступны для 80+ сетей. Мягкий лимит: ~**1 000 запросов за 10 секунд** (~100 запросов/с) на публичных эндпоинтах. При наличии бесплатного аккаунта (Freemium): **200 миллионов API-кредитов/месяц**. Для базового доступа API-ключ не требуется — наиболее простой вариант для старта и прототипирования. Формат эндпоинтов: `https://rpc.ankr.com/eth`, `https://rpc.ankr.com/bsc`, `https://rpc.ankr.com/polygon`.

#### Сводная таблица

| Провайдер | Бесплатный лимит | Нужен ключ | Сети |
|---|---|---|---|
| BscScan | 5 зап/с, 100К/сутки | Да | Только BSC |
| PolygonScan | 5 зап/с, 100К/сутки | Да | Только Polygon |
| Alchemy | 300М CU/месяц | Да | 30+ сетей |
| Infura | 100К зап/сутки | Да | 10+ сетей |
| Ankr | ~100 зап/с (публичный) | Нет (опционально) | 80+ сетей |

### 4. Руководства по Регистрации и Получению API-ключей

#### 4.1 Alchemy

1. Перейдите на [alchemy.com](https://www.alchemy.com) и нажмите **Sign Up** (в правом верхнем углу). Регистрация доступна через email или аккаунт Google.
2. Подтвердите email-адрес по ссылке из письма.
3. На экране приветствия выберите назначение — **"Infra & Tooling"** или **"Research"**. Введите название организации/проекта.
4. Вы попадёте на **Alchemy Dashboard** (dashboard.alchemy.com).
5. Нажмите **"+ Create new app"** в правом верхнем углу.
6. Введите название приложения (например, `evm-cross-linker`). Выберите нужную сеть (начните с **Ethereum Mainnet**). Нажмите **Create App**.
7. В дашборде приложения нажмите **"API Key"** (или иконку копирования рядом с URL эндпоинта). Скопируйте строку ключа — она выглядит как `abc123def456...`.
8. Повторите шаги 5–6 для дополнительных сетей (Polygon, BSC) — можно создать отдельные приложения или использовать один ключ с разными URL эндпоинтов.
9. Формат эндпоинта: `https://eth-mainnet.g.alchemy.com/v2/ВАШ_API_КЛЮЧ`

Для Python: `pip install web3`, затем `Web3(Web3.HTTPProvider("https://eth-mainnet.g.alchemy.com/v2/ВАШ_КЛЮЧ"))`.

#### 4.2 Infura

1. Перейдите на [infura.io](https://www.infura.io) и нажмите **Sign Up**.
2. Введите имя, email и пароль. Нажмите **Sign Up**.
3. Проверьте почту и подтвердите адрес по ссылке из письма.
4. На странице приветствия выберите роль и размер организации (для учебных целей — любой вариант).
5. В диалоге **"Create your first API key"** введите название проекта (например, `evm-research`). Нажмите **Submit**.
6. Infura автоматически создаст первый ключ с названием **"My First Key"**. Перейдите в **Dashboard → My First Key**.
7. Во вкладке **"All Endpoints"** все поддерживаемые сети включены по умолчанию. Скопируйте **API Key** в верхней части страницы.
8. Формат эндпоинтов: Ethereum — `https://mainnet.infura.io/v3/ВАШ_КЛЮЧ`. Polygon — `https://polygon-mainnet.infura.io/v3/ВАШ_КЛЮЧ`. BSC — `https://bsc-mainnet.infura.io/v3/ВАШ_КЛЮЧ`.

Для Python: аналогичный `Web3(Web3.HTTPProvider(...))`, как и для Alchemy.

#### 4.3 Ankr

**Вариант A — Без регистрации (публичный RPC, рекомендован для прототипирования):**
Регистрация не нужна. Используйте публичные эндпоинты напрямую:
- Ethereum: `https://rpc.ankr.com/eth`
- BSC: `https://rpc.ankr.com/bsc`
- Polygon: `https://rpc.ankr.com/polygon`

Работают сразу без ключа. Лимит ~100 запросов/с.

**Вариант B — Бесплатный аккаунт для получения ключа (рекомендован для длительной работы):**
1. Перейдите на [ankr.com](https://www.ankr.com) и нажмите **Get Started** или **Log In → Sign Up**.
2. Зарегистрируйтесь через email или OAuth (Google/GitHub).
3. Подтвердите email.
4. В дашборде Ankr перейдите в **RPC Service → Advanced API**.
5. Ваш API-ключ (токен) отображается на странице [ankr.com/rpc/advanced-api](https://www.ankr.com/rpc/advanced-api). Скопируйте его.
6. Формат аутентифицированного эндпоинта: `https://rpc.ankr.com/eth/ВАШ_API_КЛЮЧ`

#### 4.4 BscScan API

1. Перейдите на [bscscan.com](https://bscscan.com) и нажмите **Sign In → Register** в правом верхнем углу.
2. Введите имя пользователя, email, пароль. Пройдите CAPTCHA. Нажмите **Create Account**.
3. Подтвердите email по ссылке из письма.
4. После входа перейдите на [bscscan.com/myaccount](https://bscscan.com/myaccount).
5. В левом сайдбаре нажмите **API Keys**.
6. Нажмите **Add** для создания нового ключа. Введите название приложения (например, `evm-research`). Нажмите **Continue**.
7. Ключ отобразится в списке. Скопируйте его.
8. Формат запроса: `https://api.bscscan.com/api?module=...&apikey=ВАШ_КЛЮЧ`

#### 4.5 PolygonScan API

1. Перейдите на [polygonscan.com](https://polygonscan.com) и нажмите **Sign In → Click to sign up**.
2. Зарегистрируйтесь: введите имя пользователя, email, пароль. Пройдите CAPTCHA.
3. Подтвердите email.
4. После входа нажмите на имя пользователя (правый верхний угол) → **My Profile → API Keys** в сайдбаре.
5. Нажмите **Add** → введите название приложения → **Create New API Key**.
6. Скопируйте ключ из списка.
7. Формат запроса: `https://api.polygonscan.com/api?module=...&apikey=ВАШ_КЛЮЧ`

Примечание: BscScan и PolygonScan используют ту же инфраструктуру аккаунтов, что и Etherscan. Если у вас уже есть аккаунт Etherscan, один API-ключ можно использовать для всех трёх обозревателей через унифицированный эндпоинт Etherscan API v2.

### 5. План Завершения Проекта — Пошагово по Фазам

#### Фаза 0 — Конфигурация и Хранилище

Определить `config.py` с: API-ключами каждого провайдера, целевыми сетями (ETH, BSC, Polygon), URL RPC-эндпоинтов, временем блока по каждой сети, временными окнами в количестве блоков, путями к директориям локального кэша. Развернуть локальное хранилище — SQLite или файлы Parquet — для сохранения полученных данных блоков и транзакций, чтобы повторные запуски не повторяли API-запросы.

#### Фаза 1 — Выборка Блоков и Извлечение Транзакций

Для каждой сети выбрать общую точку отсчёта по времени (например, конкретный Unix timestamp). Преобразовать временные окна (1 / 10 / 100 дней) в диапазоны номеров блоков с использованием времени блока каждой сети. Загружать каждый блок с полными транзакциями (`eth_getBlockByNumber` с `hydrated=True`). Из каждой транзакции извлекать: `from`, `to`, `value`, `nonce`, `hash`, `blockNumber`, `blockTimestamp`. Сохранять сырые данные локально, с кэшированием по номеру блока на каждую сеть. Использовать асинхронную пакетную загрузку для максимизации пропускной способности в пределах лимитов API.

#### Фаза 2 — Фильтрация EOA

Для каждого уникального адреса (как `from`, так и `to`) вызвать `eth_getCode(address, "latest")`. Если результат `"0x"` (пустой) — это **EOA**, сохранить. Если возвращается байткод — это **смарт-контракт**, отбросить. Кэшировать результаты: каждый адрес нужно проверять лишь однократно по всем блокам. Это наиболее затратный по API шаг — выполнять с асинхронным параллелизмом для соблюдения лимитов.

#### Фаза 3 — Классификация Активных / Пассивных Кошельков

На основе отфильтрованного набора EOA классифицировать каждый адрес по каждой сети:

**Активный кошелёк** = адрес присутствует как `from` хотя бы в одной транзакции И имеет nonce ≥ 1. Такие адреса тратили газ и инициировали переводы.

**Пассивный кошелёк** = адрес фигурирует только как `to` и ни разу как `from` в рамках выборки (nonce = 0). Это адреса-получатели: холодное хранение, депозитные адреса, новые кошельки.

Результат: два множества на каждую сеть — `active[сеть]`, `passive[сеть]`.

#### Фаза 4 — Кросс-чейн Сопоставление Адресов

Вычислить пересечения множеств для всех парных и тройных комбинаций:

- `active[ETH] ∩ active[BSC]`
- `active[ETH] ∩ active[POLYGON]`
- `active[BSC] ∩ active[POLYGON]`
- `active[ETH] ∩ active[BSC] ∩ active[POLYGON]`

То же — для пассивных множеств. Также вычислить смешанные: активен в одной сети, пассивен в другой.

Вычислить **сходство Жаккара** для каждой пары: `|A ∩ B| / |A ∪ B|`. Сравнить наблюдаемое пересечение с ожидаемым случайным: в адресном пространстве ~2¹⁶⁰ возможных Ethereum-адресов случайное совпадение близко к нулю — любое нетривиальное пересечение статистически значимо по умолчанию.

#### Фаза 5 — Извлечение Признаков для Пересекающихся Адресов

Для каждого адреса, присутствующего в 2+ сетях, вычислить признаки на уровне адрес × сеть: суммарное число отправленных транзакций, суммарное число полученных транзакций, суммарный объём отправленных средств (в нативном токене), суммарный объём полученных средств, первый и последний timestamp активности, число уникальных контрагентов, средняя сумма транзакции, частота транзакций (txns/день).

#### Фаза 6 — Корреляционный Анализ

**6а. Временна́я корреляция:** Для адресов, активных в нескольких сетях, вычислить временной разрыв (Δ) между первой транзакцией в сети A и сети B. Построить распределение Δ. Если распределение кластеризовано в малом диапазоне (например, в пределах нескольких часов) — это признак скоординированного кросс-чейн поведения.

**6б. Объёмная корреляция:** Для пересекающихся активных адресов вычислить корреляцию Пирсона и Спирмена между объёмом_отправлено в сети A и в сети B. Высокая положительная корреляция означает, что пользователи перемещают пропорциональные суммы между сетями.

**6в. Корреляция частоты активности:** Сравнить суточные счётчики транзакций для одного адреса в разных сетях. Вычислить Pearson r по этим временным рядам для каждого адреса, затем усреднить по всем пересекающимся адресам.

**6г. Анализ соотношения активных/пассивных кошельков:** Какая доля активных кошельков сети A также присутствует в сети B (активно или пассивно)? Выше ли эта доля, чем случайное совпадение? Вычислить базовую нулевую модель и сравнить.

#### Фаза 7 — Статистическое Тестирование

**Критерий χ² (хи-квадрат):** Построить таблицы сопряжённости 2×2 (присутствует/отсутствует в сети A × присутствует/отсутствует в сети B). Вычислить χ² и p-значение. p < 0.05 подтверждает неслучайное совместное присутствие.

**Перестановочный тест:** Случайно перетасовать множества адресов 1 000 раз. Каждый раз пересчитывать сходство Жаккара для построения нулевого распределения. Сравнить наблюдаемый Jaccard с 95-м процентилем нулевого распределения.

**Pearson/Spearman:** Применить к векторам признаков на уровне адреса по всем сетям для всех пересекающихся адресов.

Все тесты выполняются для каждого размера окна (1 / 10 / 100 дней). Отчитаться о том, как значимость и эффект меняются с увеличением окна.

#### Фаза 8 — Визуализация

Подготовить: диаграммы Венна (или UpSet plots) пересечений множеств адресов; тепловые карты попарных значений Жаккара и коэффициентов Пирсона; гистограммы распределения временного разрыва Δ между первой кросс-чейн активностью; диаграммы рассеяния объёма_отправлено в сети A против сети B; столбчатые диаграммы числа активных/пассивных кошельков по сетям и объёмов пересечений; сводные таблицы результатов всех статистических тестов с p-значениями.

#### Фаза 9 — Заключение

Если пересечение значимо превышает случайное и поведенческие признаки коррелируют между сетями → **гипотеза подтверждена**: пользователи EVM повторно используют одни и те же EOA-адреса в разных сетях с устойчивыми поведенческими паттернами, что указывает на целенаправленную кросс-чейн активность. Интерпретация вероятных паттернов: арбитраж, миграция ликвидности, обфускация или многосетевые yield-стратегии.

Если пересечение незначительно → **гипотеза не подтверждена**: пользователи не склонны к повторному использованию адресов в разных EVM-сетях в рамках выборки, что свидетельствует о преобладании сетеспецифичного управления кошельками.

Оба результата валидны и должны быть отражены в отчёте с полной статистикой и графиками.

### 6. Стратегии Разделения Работы по Направлениям Разработки

Проект делится по принципу **что именно пишет каждый участник**, а не по тому, какую сеть он обрабатывает. Предлагается три стратегии.

#### Стратегия А — Разделение по Слоям Пайплайна (рекомендуется)

Каждый участник отвечает за один горизонтальный слой всего пайплайна, применительно ко всем сетям.

**Участник 1 — Слой Данных:** Пишет `api_client.py` и `sampling.py`. Отвечает за все RPC-вызовы, логику ограничения частоты запросов, асинхронную пакетную загрузку, логику повторных попыток (retry), локальное кэширование (SQLite/Parquet) и фильтрацию EOA от смарт-контрактов через `eth_getCode`. Сдаёт чистые сохранённые датасеты EOA-транзакций по каждой сети. Этот участник также настраивает API-ключи и систему конфигурации.

**Участник 2 — Слой Классификации и Сопоставления:** Берёт сохранённые датасеты от Участника 1. Пишет `classify.py` и логику кросс-чейн сопоставления. Реализует классификацию активных/пассивных кошельков (по nonce), строит множества адресов по сетям, вычисляет все пересечения, значения Жаккара и вектора признаков для пересекающихся адресов.

**Участник 3 — Слой Анализа и Визуализации:** Берёт классифицированные и сопоставленные данные от Участника 2. Пишет корреляционный анализ (временной, объёмный, частотный), все статистические тесты (χ², перестановочный) и все графики (`plots.py`). Формирует итоговые сводные таблицы и результаты, готовые для отчёта.

Между слоями есть чёткие точки передачи данных (локальные файлы / таблицы базы данных), что позволяет работать независимо и параллельно после получения результатов первого слоя.

#### Стратегия Б — Разделение по Функциональным Модулям

Каждый участник отвечает за отдельную функциональную область во всём стеке.

**Участник 1 — Инфраструктура и API:** Конфиг, API-клиенты для всех трёх провайдеров (Alchemy, Ankr, BscScan/PolygonScan), загрузка блоков, извлечение транзакций, слой хранения. Также отвечает за фильтрацию EOA от смарт-контрактов.

**Участник 2 — Аналитика Адресов:** Классификация активных/пассивных, анализ nonce, дедупликация адресов, кросс-чейн операции с множествами (пересечения, Жаккар), вычисление вектора признаков на уровне адреса.

**Участник 3 — Статистический Движок и Вывод:** Корреляции Пирсона/Спирмена, χ², перестановочные тесты, все визуализации (matplotlib/seaborn), сравнение по размерам окон (1/10/100 дней), итоговые таблицы отчёта.

#### Стратегия В — Разделение по Временным Окнам (наименее рекомендуется, но возможно)

Каждый участник обрабатывает весь пайплайн целиком, но для одного временного окна.

**Участник 1 — Окно 1 день:** Запускает полный пайплайн (загрузка → фильтрация → классификация → сопоставление → анализ) на дневном окне для всех трёх сетей.

**Участник 2 — Окно 10 дней:** Тот же полный пайплайн для 10-дневного окна.

**Участник 3 — Окно 100 дней:** Тот же полный пайплайн для 100-дневного окна.

В конце все трое объединяют результаты для сравнения того, как корреляции меняются с увеличением окна.

Недостаток Стратегии В — дублирование кода: все трое фактически пишут одно и то же. Подходит, если команда хочет максимальной независимости, но менее эффективна.

**Рекомендация: Стратегия А.** Чёткие интерфейсы между слоями, реальный параллелизм, отсутствие дублирования кода, и каждый участник развивает глубокую экспертизу в своей области.

## ЧАСТЬ II — Рекомендации по Ускорению Python-кода

Проект предполагает загрузку и обработку потенциально миллионов транзакций по трём сетям и трём временным окнам. Наивный последовательный Python окажется слишком медленным. Следующие рекомендации упорядочены от наиболее простых в реализации к наиболее продвинутым.

---

### 1. Асинхронный I/O для API-вызовов — `asyncio` + `aiohttp`

**Почему:** API-вызовы являются I/O-bound операциями. Последовательные вызовы через `requests` ждут ответа сервера перед отправкой следующего запроса. С `asyncio` можно отправлять сотни запросов параллельно в одном потоке, ограниченных только лимитом API, а не сетевой задержкой.

**Что использовать:** `aiohttp` для асинхронного HTTP, `asyncio.Semaphore` для соблюдения лимитов API.

```python
import asyncio, aiohttp

async def fetch_block(session, url, block_number, semaphore):
    async with semaphore:
        payload = {
            "jsonrpc": "2.0", "method": "eth_getBlockByNumber",
            "params": [hex(block_number), True], "id": block_number
        }
        async with session.post(url, json=payload) as resp:
            return await resp.json()

async def fetch_blocks(rpc_url, block_numbers, max_concurrent=50):
    semaphore = asyncio.Semaphore(max_concurrent)
    async with aiohttp.ClientSession() as session:
        tasks = [fetch_block(session, rpc_url, bn, semaphore) for bn in block_numbers]
        return await asyncio.gather(*tasks)
```

**Ожидаемое ускорение:** 20–100× по сравнению с последовательным `requests` в зависимости от задержки сети и уровня параллелизма. Это наиболее высокоэффективная оптимизация в данном проекте.

---

### 2. Пакетные JSON-RPC Запросы

**Почему:** Большинство EVM RPC-провайдеров поддерживают отправку нескольких JSON-RPC вызовов в одном HTTP-запросе (batch RPC). Вместо 100 запросов отправляется 1 запрос со списком из 100 вызовов. Это резко снижает накладные расходы на каждый вызов.

```python
def build_batch(block_numbers):
    return [
        {"jsonrpc": "2.0", "method": "eth_getBlockByNumber",
         "params": [hex(bn), True], "id": bn}
        for bn in block_numbers
    ]

import requests
response = requests.post(rpc_url, json=build_batch(range(1000, 1100)))
blocks = response.json()  # список из 100 результатов
```

**Ожидаемое ускорение:** Снижение накладных расходов API в 5–20×. Комбинировать с `aiohttp` для максимальной пропускной способности — отправлять крупные асинхронные батчи.

---

### 3. Polars вместо Pandas для Обработки Данных

**Почему:** После сбора данных у вас будут DataFrame с миллионами строк (транзакции), по которым необходимо выполнять groupby, операции с множествами и JOIN-ы. Pandas однопоточный и написан на Python. Polars построен на Rust, автоматически использует все ядра CPU, а его движок ленивых вычислений оптимизирует планы запросов.

**Измеренное ускорение:** Polars обычно в 3–10× быстрее Pandas на ETL-нагрузках с большими DataFrame. Для 100-дневных окон с миллионами транзакций это существенно.

```python
import polars as pl

df = pl.read_parquet("transactions_eth.parquet")

active = (
    df.filter(pl.col("nonce") >= 1)
      .filter(pl.col("is_eoa") == True)
      .select("from_address")
      .unique()
)
```

**Рекомендация:** Использовать Polars для всех операций с DataFrame. Библиотека имеет Pandas-совместимый API (`pl.from_pandas()`), поэтому миграция существующего кода на Pandas проста.

---

### 4. DuckDB для Операций с Множествами и Кросс-чейн JOIN-ов

**Почему:** Кросс-чейн сопоставление адресов (Фаза 4) — это по сути операции SQL JOIN / INTERSECT над большими множествами. DuckDB — встроенная аналитическая база данных (работает внутри процесса, без сервера), чрезвычайно быстрая для таких запросов и умеет читать Parquet-файлы напрямую.

```python
import duckdb

result = duckdb.query("""
    SELECT a.from_address
    FROM eth_active a
    INNER JOIN bsc_active b ON a.from_address = b.from_address
    INNER JOIN polygon_active p ON a.from_address = p.from_address
""").df()
```

DuckDB обрабатывает сотни миллионов строк в секунду на обычном ноутбуке. Для операций пересечения множеств и JOIN-ов признаков в данном проекте — это правильный инструмент.

---

### 5. Multiprocessing для CPU-bound Вычислений

**Почему:** Статистические вычисления (перестановочные тесты с 1 000 итераций, вычисление корреляции по миллионам пар адресов) являются CPU-bound. GIL Python блокирует настоящую многопоточность, но `multiprocessing` запускает отдельные процессы, каждый из которых получает полное ядро CPU.

```python
from multiprocessing import Pool

def compute_jaccard(pair):
    set_a, set_b = pair
    intersection = len(set_a & set_b)
    union = len(set_a | set_b)
    return intersection / union if union else 0

with Pool(processes=4) as pool:
    results = pool.map(compute_jaccard, address_set_pairs)
```

**`aiomultiprocess`** объединяет асинхронный I/O с multiprocessing — полезно для одновременного выполнения async API-вызовов в нескольких процессах. Хорошо подходит для шага фильтрации EOA, когда нужно запросить `eth_getCode` для миллионов адресов.

---

### 6. Numba для Пользовательских Числовых Циклов

**Почему:** При написании пользовательских числовых вычислений — например, распределений временны́х разрывов, скользящей корреляции по временному ряду или внутреннего цикла перестановочного теста — обычные циклы Python крайне медленны. Numba JIT-компилирует эти циклы в нативный машинный код при первом вызове.

```python
from numba import njit
import numpy as np

@njit
def compute_deltas(timestamps_a, timestamps_b):
    result = np.empty(len(timestamps_a))
    for i in range(len(timestamps_a)):
        result[i] = timestamps_b[i] - timestamps_a[i]
    return result
```

**Лучшие случаи применения:** Внутренний цикл перестановочного теста (тысячи раз перемешивать массивы и считать пересечения), вычисление временны́х разрывов по миллионам пар адресов, скользящая статистика частоты.

**Не полезно для:** Операций Pandas/Polars (они уже оптимизированы), API-вызовов (I/O-bound, используйте asyncio).

---

### 7. cuDF / RAPIDS — GPU-ускорение (Опционально, Продвинутый Уровень)

**Почему:** При наличии NVIDIA GPU библиотека RAPIDS (`cuDF`) предоставляет Pandas-совместимый API для работы с DataFrame, выполняя все операции на GPU. На этапах извлечения признаков и вычисления корреляций GPU-ускорение может дать 10–50× прирост скорости по сравнению с Pandas на CPU.

```python
import cudf  # требует NVIDIA GPU + CUDA

df = cudf.read_parquet("transactions_eth.parquet")
grouped = df.groupby("from_address")["value"].sum()
```

**Реалистичная оценка:** Для учебного проекта cuDF практичен только при наличии NVIDIA GPU (например, Google Colab Pro с A100 или локальная рабочая станция). В противном случае стека Polars + DuckDB + Numba достаточно, и он проще в настройке.

---

### 8. Локальное Кэширование и Хранение в Parquet

**Почему:** Повторная загрузка одних и тех же блоков при каждом запуске расточительна и быстро исчерпает бесплатные лимиты API. После первой загрузки сохранять все данные в Parquet-файлы (сжатое колоночное хранилище). Все последующие запуски читают с диска — что на порядки быстрее API-вызовов.

```python
import polars as pl

# Сохранить после загрузки
df.write_parquet("blocks_eth_1day.parquet", compression="zstd")

# Загружать при последующих запусках
df = pl.read_parquet("blocks_eth_1day.parquet")
```

Parquet со сжатием `zstd` обычно в 5–10× меньше CSV и в 10–50× быстрее читается.
